# Spotify Lyrics Widget — Roadmap (current)

Date: 2026-05-25
Last updated: 2026-06-29

**Trunk status:** `master` contains V1 through V3 portable onboarding plus the taskbar
controller lifecycle work. Branch `codex/installer` is the V3.1 Windows installer release
candidate: it adds the Inno Setup installer, installer tests, build script, and version
metadata `3.1.0`. Remaining release work is merge, external smoke testing, code signing /
antivirus false-positive handling, and deferred features.

This is the high-level roadmap. Each phase still needs its own detailed plan/spec
(brainstorm → spec → plan) in a fresh session before implementation.

## Current state (done)

- **V1** — lyrics core: PKCE auth, 1s Spotify poll, LRCLIB lookup + ranking, LRC parse,
  frameless always-on-top widget, config in `%APPDATA%`.
- **V1.1** — UI polish: rounded `#121212` panel, green border, typography, stable hover,
  long-title elide.
- **V1.2** — stability (commit `56ed38d`, tested):
  - Logging to `%APPDATA%/spotify-lyrics-widget/widget.log` + `QMessageBox.critical` on
    fatal/startup/auth errors; `run.pyw` (pythonw, no console).
  - Fixed geometry (no title/lyric jump, 2-line lane, 2nd line never clipped), correct
    first frame.
  - Offline indicator as overlay (does not push content).
  - Single-instance guard (`QLocalServer`/`QLocalSocket`; 2nd launch raises the 1st).
  - 429 backoff: reads `Retry-After`, stops polling for that window, UI shows
    "rate limited - retrying", resumes. Fixes the hammering that caused the real ~15h ban.
- **V1.3** — reachability + hotfixes (merged to master through `c164cb0`):
  - System tray icon: green circle, left-click raises widget, right-click menu
    (Hide/Show widget, Open log file, Quit), clean exit removes icon.
  - Start-menu / desktop shortcut installer (`scripts/install_shortcuts.py`).
  - Offline status moved into lyric lane (removed right-top overlay).
  - **Hotfix: font crash** — 原計畫用 bundled NotoSansTC-VF.ttf (36 MB variable font)，
    但該字體在 Qt 6.11.0 的 `QFontDatabase.addApplicationFont` 會觸發 access violation，
    殺死 UI 主線程。改用系統字體 Microsoft JhengHei UI（微軟正黑體），同樣支援中文。
    字體檔案仍留在 `assets/fonts/` 但不再被載入。
  - **Hotfix: widget visual refresh / render path** — Qt frameless translucent window
    (`WA_TranslucentBackground`) 在 Windows 上有時不會把已更新的 label 像素送到
    layered window。修正是移除 `WA_TranslucentBackground`，改用一般不透明 top-level window，
    避免走 layered alpha compositing。（V1.3 當時用 `QRegion` mask 做圓角；**post-V1.3 已改用
    Windows 11 DWM 圓角＋綠邊框取代 mask**，見下方 Post-V1.3 stability 與 design spec。）
  - Crash logging: `sys.excepthook` 導入 log file，不再遺失 pythonw 下的未捕獲例外。
  - UI heartbeat log（每 30 秒）、track change slot log、shutdown log，方便診斷。
  - Spotify playback summary log（每次 poll 記錄 track/artist/progress）。
- **Post-V1.3 stability (master, 2026-05-27 session):**
  - **Decoupled progress from lyrics + clear stale lyrics on track change** (`84e2105`):
    progress bar now advances whenever playback is active even with no/unavailable lyrics
    (it used to freeze because `_on_ui_tick` returned early on empty `_lyrics`); and the
    previous track's lyrics are cleared on switch so they can't be matched against the new
    progress during the async lookup gap (the "title is new but lyric is the old song's last
    line" symptom).
  - **LRCLIB request timeout 5s → 10s** (`6dd5491`): LRCLIB had slowed to ~6-9s, turning every
    lookup into a transient "unavailable". NOTE: the V1.4 NetEase fallback does NOT cover this
    — a timeout is *unavailable*, not a confirmed miss, so it never triggers NetEase. V2.02
    later adds a one-shot NetEase salvage path for this case without writing no-lyrics cache.
  - **Window corners → Windows 11 DWM**: replaced the jagged `QRegion` mask with DWM rounded
    corners + a DWM green system border (`DWMWCP_ROUND` + `DWMWA_BORDER_COLOR`). Opaque window,
    smooth + stable; the panel draws no border/radius of its own. Win11-only (guarded no-op on
    older Windows → square, borderless).
- **V1.4** — LRCLIB contract hardening + gated NetEase fallback (branch
  `feature/v1-4-netease-fallback`, final implementation commit `2b26a26`; verification
  recorded 2026-05-28):
  - LRCLIB transient failures are treated as unavailable, not confirmed misses; NetEase only
    runs on confirmed LRCLIB miss.
  - NetEase fallback ranks multiple candidates, requires timed lyrics, normalizes
    Traditional/Simplified Chinese via `zhconv`, filters NetEase credit lines, and applies a
    cooldown brake on NetEase-unavailable.
- **V1.5** — Logging hygiene (commits `ee239f2`, `38cfea7`, `c80ccc8`, `4318eff`):
  - Closes silent log gaps in `lyrics_worker.run()`, LRCLIB `/get` and `/search`, token
    pre-refresh in `App._ensure_auth`, and Spotify network exceptions in `_poll_once`.
  - No behaviour change; additive logs only. Automated verification: full suite
    `161 passed` on 2026-05-28 with PyQt6 6.11.0 / Qt runtime 6.11.1.
- **V2** — Playback controls + title marquee (feature commits `f63e398`, `1aa19ae`,
  `d85fe42`, `61ededb`, `3f8c793`; final layout/style polish through `3fc256b`):
  - Added hover-only previous/play-pause/next controls wired to Spotify playback endpoints.
  - Added one-time reauth for playback scopes and off-UI-thread playback requests with
    duplicate-drop and 429 cooldown handling.
  - Added left-aligned long-title marquee; controls live in a reserved right-side slot
    between the title area and close button, so long titles no longer collide with controls.
  - Final user-reviewed control style uses icon-only play/pause, no circular backing, with
    adjusted icon width/height for visual parity with previous/next.
- **V2.01** — Follow-up patches (commits `f749fce`, `50ba487`; verified `205 passed` on
  2026-06-04):
  - Clamp displayed lyric text to at most two visual lines using `QTextLayout`, because an
    LRCLIB timestamp line can wrap into three UI lines even without embedded newlines.
  - When `PUT /me/player/play` fails with `NO_ACTIVE_DEVICE`, fetch available Spotify
    devices and retry play once with `device_id`. This fixes the common "Spotify app is open
    but not currently playing" case after reauthorization with `user-read-playback-state`.
- **V2.02** — LRCLIB-unavailable NetEase salvage (verified `206 passed` on 2026-06-06):
  - When LRCLIB is temporarily unavailable, the worker now tries NetEase once for that track
    instead of immediately emitting unavailable.
  - If NetEase finds lyrics, the result is cached and displayed like any other lyric hit.
  - If NetEase also misses or is unavailable, the UI emits unavailable and does **not** write
    the no-lyrics cache, because LRCLIB never completed a confirmed miss.
- **V2.03** — Size presets / density presets + tray polish (merged to master, verified
  `218 passed` on 2026-06-07):
  - Tray menu `Size` submenu can switch Current (420x112), Compact (380x96), Small (340x84),
    and Mini (300x74). All presets use two visual lyric lines.
  - Transport buttons scale proportionally via painter scaling.
  - Selected preset persisted to config and restored on startup.
  - Tray simplified: left-click toggles widget visibility, right-click menu has Size + Quit
    only (removed Hide/Show and Open log file).
  - Optimistic play/pause: button icon flips immediately on click, corrected by next poll.
  - Spec: `docs/superpowers/specs/2026-06-06-spotify-widget-size-presets-design.md`.
  - Plan: `docs/superpowers/plans/2026-06-06-spotify-widget-size-presets.md`.
- **V2.04** — Three size presets (merged to master through `dce5da0`, verified
  `224 passed` on 2026-06-07):
  - Replace Current / Compact / Small / Mini with Small (300x74), Medium (360x90), Large (420x112).
  - Removed V2.03 config keys use simple aliases: mini -> small, compact -> medium,
    current -> large. Existing `small` values stay `small`.
  - Fixes Large / Medium -> Small stale layout resize by locking the fixed preset size after
    layout activation (`cacee90`).
  - Spec: `docs/superpowers/specs/2026-06-07-spotify-widget-three-size-presets-design.md`.
  - Plan: `docs/superpowers/plans/2026-06-07-spotify-widget-three-size-presets.md`.
- **V3** — Portable onboarding (merged to master through `189e99e`, verified 2026-06-08):
  - First-run onboarding dialog for Spotify Client ID setup.
  - PyInstaller one-folder portable build with `build_portable.ps1`.
  - Chinese README and packaging tests.
  - Spec: `docs/superpowers/specs/2026-06-07-spotify-widget-v3-portable-onboarding-design.md`.

## Decisions & rationale (do not revisit without a new reason)

- **No startup-on-boot** (user, 2026-05-25). If ever added, it MUST be gated behind a
  verified V1.2 single-instance guard + 429 backoff — otherwise autostart resurrects the
  zombie / multi-instance hammering that caused the ~15h 429 ban.
- **Log: kept; "Open log file" confirmed for V1.3 tray menu** (user, 2026-05-25). The
  background `widget.log` is the replacement for the hidden console (`run.pyw` = no console,
  so errors would otherwise vanish) — keep it unconditionally. V1.2 already pops
  `QMessageBox` for fatal/startup/auth errors. The only addition is an "Open log file"
  action (`os.startfile`) in the V1.3 tray right-click menu so the existing log is one click
  away. No full in-app viewer (over-engineering for a personal tool).
- **System tray icon → V1.3 ③** (Claude × Codex consensus, 2026-05-25). Same
  "visibility / reachability" category as the shortcut, not playback. Reuses the existing
  single-instance raise-window logic (`QLocalServer`/`QLocalSocket`). Two things to verify
  during implementation: (a) **Windows may block a background app from forcing its window to
  the foreground** — `show()`/`raise_()`/`activateWindow()` can just flash the taskbar
  instead of actually raising; test from a truly covered state, add a workaround if blocked.
  (b) Clean exit must remove the tray icon (no orphaned icon), and single-instance must not
  spawn a second tray icon. Icon asset path will need PyInstaller `resource_path` handling
  at V3.
- **Chinese font → Microsoft JhengHei UI (V1.3 final).** The original V1.3 plan loaded
  bundled `NotoSansTC-VF.ttf` via `QFontDatabase.addApplicationFont`, but live testing on
  Qt 6.11.0 showed an access violation that killed the UI main thread. Final implementation
  uses system font detection in this order: `Microsoft JhengHei UI`, `Microsoft JhengHei`,
  then `Segoe UI`. The Noto font file stays in `assets/fonts/` for reference/license history
  but must not be loaded by V1.3.
- **Offline status → V1.3 lyric lane.** Do not keep a right-top `offline` overlay. When
  Spotify cannot be reached, show a central `offline` status in the lyric lane. This keeps
  the top row reserved for title / future controls / close button and avoids overlap in V2.
- **V2 trims:** edge-snap CUT (nice-to-have only). Drag-to-resize-width DEFERRED to a later
  version.
- **Auto-resize REJECTED** (content-driven / adaptive). It reintroduces the title/lyric
  jumping that V1.2 Task 2 fixed. The only allowed resize is user-draggable WIDTH persisted
  to config. Do not revisit.
- **Title left-align / controls overlay** folds into V2's marquee rework (not a standalone
  version). V2 uses a left-aligned title that elides before a reserved right-side transport
  slot, while controls are absolute top-row children visible only on widget hover. Do not
  revive the old 5-50 / 60-80 / 90-95 percentage-slot layout.
- **V2 final transport style:** play/pause is icon-only, no circular backing. Previous,
  play/pause, and next should read as one compact control family, with hover-green icons.
- **V2.01 lyric clamp:** do not trust one LRC timestamp line to equal one UI line. The UI
  must clamp actual visual layout to at most two lines.
- **V2.01 not-playing playback:** if Spotify returns `NO_ACTIVE_DEVICE`, retry once with an
  available non-restricted device. If no device exists, log the concrete reason; do not
  invent a local player.
- **V2.02 lyrics fallback:** LRCLIB remains primary, and `None` still means "valid response,
  no synced lyrics." However, LRCLIB-unavailable now triggers one NetEase salvage attempt for
  that track. A NetEase hit is cached; a NetEase miss after LRCLIB-unavailable is **not**
  cached as no lyrics.
- **Playlist-add** (add current track to a playlist + remember default) is split out of V2
  into its own later phase; do controls + marquee first.
- **Lyrics fallback (V1.4):** LRCLIB stays primary; add ONE optional fallback at a time,
  behind config + clear failure handling. V1.4 choice is NetEase because it does not require
  the user's private Spotify `sp_dc` cookie. `sp_dc`-based official lyrics are rejected for
  V1.4: unofficial, expires, ToS gray area, and too fragile for the core path.
- **V1.4 revised after spike (2026-05-26, Claude × Codex, 4 rounds; do not revisit without a
  new reason):**
  - **First fix LRCLIB's contract** — `None` must mean "valid response, no synced lyrics".
    LRCLIB 429/malformed-JSON/non-200 must be *unavailable*, not a confirmed miss; the
    malformed-JSON case also fixes a silent worker-thread crash. V2.02 later relaxed the
    old "never on LRCLIB-unavailable" gate: LRCLIB-unavailable may trigger one NetEase
    salvage attempt, but still never writes no-lyrics cache unless LRCLIB completed a
    confirmed miss.
  - **NetEase endpoint spike PASSED** (real HTTP from Taiwan, no cookie). It also exposed:
    (A) `search[0]` can be an untimed cover → rank multiple candidates and require timed
    lines; (B) Traditional↔Simplified mismatch (周杰倫/周杰伦, 鄧紫棋/邓紫棋) → add `zhconv`
    (user-approved new dep) and Traditionalize NetEase lyrics for display; (C) NetEase
    prepends `作词/作曲/编曲 …` credit lines → NetEase-only filter.
  - **One unified NetEase cooldown**, not a separate circuit breaker: any
    `NeteaseUnavailableError` sets it (429 → `Retry-After`, else 60s). No consecutive-failure
    escalation — over-engineering for a single-user tool.
  - **Rejected for V1.4:** `opencc` (heavier than `zhconv` for no meaningful gain here);
    "accept best search match" without a timed-lyrics gate (wrong lyrics worse than none);
    putting the NetEase credit filter in `lrc_parser` (would pollute the LRCLIB primary path).
  - **Window-render note (final):** translucent surface (`WA_TranslucentBackground`) → interim
    `QRegion` mask (`120bfd8`/`4589a84`; stable but jagged corners) → **Windows 11 DWM rounded
    corners + DWM green system border** (`DWMWCP_ROUND` + `DWMWA_BORDER_COLOR`). Opaque window
    throughout, so no transparent-window repaint risk; DWM adds smooth corners + a single
    corner-hugging border. None of this affects V1.4 (data-layer only); a late NetEase result
    repaints via normal Qt paths.
